배경과 문제 정의

사업부의 핵심 지식이 개인과 비정형 문서에 종속되어 조직 자산으로 축적되지 못하고 있다.

현행 문제

  • 설계 노하우·판단 근거가 개인 문서·구두 지식에 분산
  • 담당자 이동 시 지식 단절, 동일 질의 반복
  • 자료 탐색 비용 과다 — 본업(설계·검증) 시간 잠식
  • 범용 AI 도구는 사내 지식에 접근 불가

두 시스템의 해소 방향

  • 지식을 구조화된 조직 자산으로 축적
  • 질의 응답을 시스템화 — 출처 기반 답변 제공
  • 사내 AI가 내부 자료를 근거로 응답
  • 탐색 시간 단축, 지식 재사용성 확보

두 시스템의 포지셔닝

AI Data Hub — 활용(소비) 계층

사내 데이터를 대상으로 한 시맨틱 검색?시맨틱 검색 (Semantic Search) — 키워드 일치가 아니라 문장의 의미를 수치화한 벡터 임베딩으로 유사도를 계산해 "뜻이 가까운" 자료를 찾는 검색.예: "강성 부족"과 "휨 과다"를 의미상 유사 자료로 연결 과 RAG?RAG (Retrieval-Augmented Generation) — AI가 답을 생성하기 전에 사내 자료를 먼저 검색해 그 근거에 기반해 답하도록 하는 방식. 출처 추적이 가능해 환각(허위 생성)을 억제한다. 기반 응답을 제공하는 계층.

MX White Paper — 정리(생산) 계층

지식을 조직 체계로 정리·연결하고 버전·권한으로 관리하는 계층. 구조화 데이터?구조화 데이터 (Structured Data, JSON Schema) — 문서를 화면 표현이 아닌 기계가 해석 가능한 정의된 데이터 형식으로 저장하는 것. 검색·내보내기·AI 자동화가 동일 데이터를 재사용할 수 있다. 로 저장해 후속 자동화의 토대를 확보한다.

핵심 — 지식을 개인 종속에서 분리하여 구조화된 조직 자산으로 전환

시스템 1 — AI Data Hub

사내 데이터를 AI가 직접 활용 가능한 형태로 통합한 데이터 인프라.

처리 흐름

1. 적재 / 표준화

Word·Excel·PPT·PDF를 표준 데이터 스키마로 자동 변환·저장.

2. 의미 색인

적재 즉시 벡터 임베딩?벡터 임베딩 (Vector Embedding) — 문장의 의미를 수백 차원의 숫자 벡터로 변환한 표현. 벡터 간 거리가 가까울수록 의미가 유사하다. 본 시스템은 다국어 임베딩 모델로 768차원 벡터를 사용한다. 생성 후 의미 색인 구축.

3. 응답

질의에 대해 적합 지식영역을 자동 선택, 근거를 인용해 답변.

시스템 강점

강점현장 효과
의미 기반 검색키워드를 모르더라도 유사 불량 사례·관련 측정 결과 탐색
출처 인용 응답생성형 환각 억제 — 어느 문서 몇 번 항목인지 추적 가능
지식영역 자동 라우팅질의만으로 참조 대상 데이터셋을 시스템이 선정
표준 프로토콜 연동MCP?MCP (Model Context Protocol) — AI 클라이언트가 외부 데이터·도구를 표준 방식으로 연결하는 개방형 프로토콜. 특정 AI 제품에 종속되지 않고 Claude·Copilot 등 다수 클라이언트가 동일하게 연동된다. 표준으로 사내 AI 도구 7종 연동
폐쇄망 운용데이터 외부 반출 없음 — 보안 요건 충족

요지 — 사내 자료를 근거로 즉답하는 데이터 활용 인프라. 과거 사례·측정·시뮬레이션 결과를 인적 의존 없이 확보.

기반 기술: pgvector(벡터 DB) · 다국어 임베딩 768차원 · MCP Streamable HTTP

시스템 2 — MX White Paper

지식을 조직 체계로 정리·연결·보존하는 지식 관리 플랫폼.

핵심 기능

정리

부문→팀→그룹→파트 계층으로 정돈. 기존 자산 대량 반입.

연결

위키링크?위키링크 / 백링크 (Wiki Link / Backlink) — 문서에서 다른 문서를 참조하면 양방향 연결이 자동 생성되는 방식. 미작성 참조는 별도 표시되어 지식 공백이 가시화된다. 로 문서 간 의존성 자동 형성·시각화.

보존

저장 단위 버전 관리·복원, 권한·감사 이력 기록.

시스템 강점

강점현장 효과
지식 연속성담당자 이동과 무관하게 백서 잔존 — 인수인계 리스크 저감
표현력표·그래프·수식·다이어그램·계산 위젯을 단일 문서에 통합
협업 안정성권한·변경이력·버전 복원으로 동시 편집 충돌 통제
자동화 대비구조화 저장으로 후속 AI 자동화를 추가 개발 없이 수용

요지 — 조직이 소유하는 살아있는 기술 백서. 인적 이탈과 무관하게 지식이 조직에 귀속.

기반 기술: 구조화 데이터(JSON Schema SSOT) · 버전·권한(RBAC) · 위키 의존성 그래프

두 시스템의 연계 구조

생산(White Paper)과 소비(Data Hub)가 결합되어 지식 품질이 자기강화되는 순환을 형성한다.

1. 생산

엔지니어가 White Paper에 노하우·결과를 구조화 정리.

2. 통합

백서·문서가 Data Hub로 수렴, 의미 색인.

3. 소비

질의 시 AI가 근거 기반 응답.

구조적 의미 — 정리된 지식의 품질이 활용 품질을 결정한다. 정리 노력이 전사 AI 응답 품질로 환원되는 자기강화 루프이며, 이 관계 자체를 구조화하는 것이 다음 장의 방향성이다.

생산(사람) → 통합(허브) → 소비(AI) → 재생산 — 자기강화 순환

방향성 ① — 업무 온톨로지화

문서 통합을 넘어, 문서가 담은 "업무 그 자체"를 관계 모델로 구조화한다.

1단계 — 업무를 정밀 기술

업무 = 프로세스 + 산출물

모든 설계 업무는 진행 프로세스(입력·방법·판단)와 산출물(설계값·성능값·판정 근거)로 분해된다. 이를 White Paper에 정형 기술한다.

2단계 — 의미 단위 분할

기술 내용을 청킹?청킹 (Chunking) — 긴 문서를 의미가 완결되는 작은 단위로 분할하는 처리. 각 청크가 검색·관계 연결의 최소 단위가 된다.예: 입력조건 / 해석방법 / 설계값 / 성능결과 / 판정근거 각각이 1개 청크 으로 분할한다 — 입력조건·해석방법·설계값·성능값·판정근거 단위.

3단계 — 관계 모델로 저장

업무 정형 기술

온톨로지?온톨로지 (Ontology) — 개념(업무·데이터)과 그 사이 관계를 형식적으로 정의한 지식 모델. 단순 분류를 넘어 "무엇이 무엇에 영향을 주는가"를 기계가 추론 가능하게 표현한다. 와 Graph DB?Graph DB (그래프 데이터베이스) — 데이터를 노드(점)와 엣지(관계 선)로 저장하는 DB. 관계를 따라가는 다단계 탐색이 표(RDB) 대비 자연스럽고 빠르다.예: A업무→(영향)→B설계값→(제약)→C성능지표 경로 추적 의 역할 — 표가 값을 칸에 담는다면, 관계 그래프는 업무·데이터를 잇는 의존성 망을 표현해 다단계 연관 추적을 가능케 한다.

목표 산출물 — 업무·데이터 간 관계가 질의 가능한 형태로 구조화

방향성 ② — 시스템 엔지니어링 최적화

관계 그래프가 확보되면 부서 간 입출력·trade-off를 연결한 전역 최적 탐색이 가능해진다.

두 시스템의 역할 분담

AI Data Hub — 설계·성능 데이터 확보

각 부서가 설계하는 부품·구조의 설계값·성능값(두께·강성·질량·원가·수명 등 실측/해석 수치)을 탐색·수집한다.

MX White Paper — 의미 관계 정의

그 데이터 간 의미 관계(인과·영향·제약)를 정의·축적한다. 예: 두께 증가 → 강성↑·질량↑·원가↑ 의 결합 관계.

왜 돌파구인가 — 국소 최적 문제

현행 — 부분 최적의 함정

부서별 개별 최적화는 부서 단위로는 우수해도 전체 최적이 아닐 수 있다. 부서 간 trade-off?Trade-off (상충 관계) — 한 지표를 개선하면 다른 지표가 악화되는 관계. 예: 경량화 → 강성 저하 / 원가 절감 → 수명 단축 다부서 설계에서는 이런 상충이 사슬처럼 얽힌다. 관계가 문서·인적 지식에 분산되어 전체를 조망하지 못하고 국소 최적?Local Optima (국소 최적) — 주변만 보면 최선이지만 전체로 보면 더 나은 해가 존재하는 지점. 부서별 개별 최적화의 합이 전체 최적과 일치하지 않을 때 발생한다. 에 고착된다.

지향 — 전역 최적 탐색

관계 그래프가 부서 간 입출력 데이터와 trade-off 사슬을 단일 망으로 연결. 시스템 엔지니어링?시스템 엔지니어링 (Systems Engineering) — 개별 부품이 아니라 시스템 전체의 입출력·제약·목표를 통합적으로 다루어 최적해를 도출하는 공학 방법론. 관점에서 연관 업무·데이터를 자동 도출, 부서를 횡단하는 최적안을 탐색하여 국소 최적을 탈출한다.

설계 관점 비유 — 부서별 최적화는 각 봉우리의 정점 도달이나 전역 최댓값(진정한 정상)은 다른 봉우리일 수 있다. 관계 그래프는 봉우리를 잇는 능선 정보를 제공하여 전역 최적해 경로를 가시화한다.

목표 — 부서 횡단 trade-off 최적안 탐색으로 개발의 국소 최적 돌파
