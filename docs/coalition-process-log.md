## 4-3. 💬 의견 교환 (9라운드, 전문 발언)

**1 · 개시 · pwr-cell-design**
> "'층을 늘리면 용량이 는다'는 절반만 맞다. 셀 부피가 고정이면 층마다 집전체·분리막이라는 비활성 두께가 고정비로 붙어 이득이 선형이 아니다. 그리고 로딩(두꺼운 전극)↑는 에너지밀도는 올리지만 충전 율특성·리튬 석출 마진은 악화다 — 방향은 확정적이다." [PWR-093·100·091]

**2 · 보강 · pwr-cell-chemistry**
> "근본적으로 에너지밀도를 올리는 수단은 전압↑·소재(실리콘)↑·비활성↓ 셋뿐이다. 층수는 '비활성/부피' 축을 건드리는 것. 실리콘을 더 넣어 층당 용량을 키우려 하면 체적변화·SEI 소모·수명의 벽에 막힌다." (↳ 구조의 "비활성 비율"에 소재 축 한계 추가) [PWR-061·078·072]

**3 · 보강 · sim-battery-electrochem**
> "두꺼운 전극이 왜 위험한지 물리로 말하겠다. 확산 경로가 길어져 국부 음극 전위가 내려가 리튬 석출이 시작된다. 판정은 평균이 아니라 분리막 계면 최악점이고 — 정상 전압 범위 안에서도 저온·고SOC·고율이면 난다. 셀 외부 전압엔 안 보인다." (↳ 설계의 "율특성 악화"를 음극 전위 물리로 근거화) [SIM-123·139·131]

**4 · 끼어듦 · pwr-charging-algorithm**
> "충전 관점을 더하겠다. 용량↑에 충전 시간을 유지하려면 전류↑, 비가역 발열은 전류의 제곱이다. 게다가 두꺼운 전극은 수용 전류가 낮아 같은 C-rate라도 음극 전위 마진이 줄어 충전이 느려지거나 석출한다. 핵심은 충전기 W가 아니라 음극 전위 마진이다." (↳ 해석의 "석출"을 충전 프로파일 제약으로 연결) [PWR-036·048·038]

**5 · 끼어듦 · pwr-swelling**
> "전기화학만 문제가 아니다. 층·두께↑ = 스웰링 팽창력↑. 설계 갭을 소진하면 팽창력이 디스플레이 배면으로 전달돼 OLED 무라·디스플레이 들뜸·백커버 벌어짐이 난다. 이건 '두께 몇 mm'가 아니라 '힘 몇 N이 어디에 걸리나'의 문제다." (↳ 전기·충전 축에 기구(세트 변형) 축 추가) [PWR-514·525]

**6 · 반박 · pwr-pack-integration**
> "'그럼 스웰링 갭을 키우면 되지 않나?' — 안 된다. 갭 확대와 낙하 신뢰성은 정면 충돌이다. 갭이 크면 낙하 시 셀이 유동해 관성 하중이 탭으로 흘러 탭이 파단한다. 갭은 '여유 있으면 좋은 것'이 아니라 EOL 성장 + 공차 + 처짐의 최소 필요량만." (↳ 스웰링의 "갭으로 수용" 제안을 낙하 근거로 반박) [PWR-385·366·387]

**7 · 끼어듦 · pwr-thermal-runaway**
> "가장 무거운 축을 빼먹었다. 용량/에너지↑는 열폭주 시 총 방출 에너지·벤트 가스량↑ = 세트 전파·화염 심각도↑다. 가연 가스 + 셀 자체 산소 방출로 자립 연소하므로 외부 산소를 끊어도 안 멈춘다. 안전 인증 마진을 직접 잠식한다." (↳ 성능·기구 논의에 "사고 시 심각도" 축을 세움) [PWR-427·431·421]

**8 · 보강 · sim-battery-abuse**

## 협진 방식 (공통 규약)

```
질문 → recommend_agents(자연어)         # 전문가 자동 선별 (score 랭킹 + why)
     → agent_search(agent_type, q, mode) # 각 전문가 지식카드 검색, retrieval_config 자동 적용
                                          # (mode: semantic | fts | hybrid | tag)
     → 종합 / 교환(반박·중재·판정)         # 클라이언트 LLM 책임 (서버는 stateless)
```

- **가드레일** — 반환 score 가 threshold(0.3, agent_record_scope 30) 미달이면 refused=true 로 답변 거부. 추측 안 함. (1번 폴더블 1차 질의 5/5 거부가 실례)
- **sample 정렬** — 모호 질의는 거부되므로, 각 전문가 sample 표현에 맞춘 질의라야 근거를 반환한다.
- **인용** — 모든 소견은 record id (예: DOC-CCT-REL-2026-0000000373)로 출처를 남긴다.
- **협진 주체** — MCP 서버는 stateless. 반박·중재·판정의 오케스트레이션은 클라이언트 LLM(=이 세션)이 수행하고, 서버는 근거(record)와 규약만 제공한다.
