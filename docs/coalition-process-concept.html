협진은 어떻게 작동하는가

질문 하나를 도메인 전문가 에이전트에게 자동 배분하고, 각자 자기 지식카드에서만 근거를 찾아 답하며, 소견들이 서로 반박·중재·판정하도록 엮는다. 근거에 묶인 다관점 진단.

205 전문가 에이전트각 ~30 지식카드서버 stateless지휘 = 클라이언트 LLM모든 소견 record 인용

파이프라인 질문 → 소견까지 4단계 · 앞 3단계는 MCP 도구, 마지막은 클라이언트 LLM

1

질문

증상·구조·조건 그대로 자연어.

자연어 입력
2

전문가 자동 선별

적합 전문가를 score로 랭킹 + why(sample).

recommend_agents(q)
3

근거 검색

각 전문가 카드 검색, retrieval_config 자동.

agent_search(type, q, mode)
4

교환 · 종합

개시→반박→중재→판정으로 엮음.

클라이언트 LLM

핵심 개념

전문가 = RAG 페르소나PERSONA
지식카드(expert_knowledge_card) + system_prompt(역할·인용·거부 규칙). get_agent_session이 채택.
자동 선별ROUTING
질문을 전문가 sample과 견줘 랭킹. 왜 뽑혔는지 근거 동봉 — 블랙박스 아님.
근거 검색RETRIEVAL
전문가별 retrieval_config(top_k·threshold·scope) 자동. mode = semantic·fts·hybrid·tag.
인용CITATION
모든 주장에 record id (DOC-CCT-REL-…373 §증상). 근거 없는 문장 불가.
statelessORCHESTRATION
서버는 근거·규약만. 반박·중재·판정의 지휘는 클라이언트 LLM.

가드레일 환각 방지

거부 · 모호 질의
"힌지 크랙 원인 알려줘"

refused — score < 문턱(≈0.3), 추측 안 함. 실제 1차 모호 질의는 5전문가 전원 거부.

근거 · 정렬 질의
"폴딩 20만회 상온 통과, -20℃서 UTG 크랙"

grounded — sample 어휘 정렬로 score 0.95, 증상·원인·대책 반환.

교환 모델 각자 말하고 끝이 아니라 서로에게 반응 · 고정 대본 아님

개시OPENING

첫 전문가가 자기 근거로 진단을 연다 (대개 교과서적 전제).

반박REBUT

다른 전문가가 그 전제를 자기 근거로 되친다.

중재MEDIATE

두 관점을 상위 원리로 수렴 — "같은 현상의 다른 축".

끼어듦INTERJECT

인접 도메인이 빠진 축(낙하·기구·안전)을 추가.

판정VERDICT

검증·FA가 증거로 매듭 — "추정 그만, 파면부터".

개시→ 반박→ 중재→ 끼어듦→ 판정 "제안→기각" 라운드가 끼거나 9전문가로 길어지기도.

왜 교환인가

병렬 · 독립 의견

첫 단정에서 멈춘다

각자 한 번씩 말하고 끝. 반례가 나란히 놓일 뿐 부딪지 않아 성급한 대책이 안 걸러진다.

교환 · 서로 반응

한 도메인 대책 = 다른 도메인 새 문제

"언더필 빼자"→낙하·CPI 근거로 기각, "갭 키우자"→낙하 반박. 잘못된 결정을 실행 전에 거른다.

파이프라인 recommend_agents → agent_search → 교환·종합 · 서버 stateless, 지휘는 클라이언트 LLM. 실제 4건 전체 기록은 협진 과정 로그. · 상단 THEME 전환.
