왜 이걸 해야 하는가

사업부 엔지니어가 매일 작성하는 문서가 AI에게는 닫혀 있다. 사내 AI가 사업부 도메인 지식을 모르는 상태로 운영되고 있다.

현재 손실

사내 AI가 사업부 지식을 모름

Cline SR은 사업부 매뉴얼·해석 보고서·재료 사양을 학습하지 않은 상태로 동작.

같은 질문에 사람이 반복 응답

"IGA offset 계산법", "재료 물성 기준" 같은 도메인 질문이 동일하게 반복. 답은 문서에 이미 있으나 AI가 못 읽음.

지식 축적 실패

퇴직·이동 시 도메인 지식이 함께 사라짐. 문서로 남겨도 검색이 어려워 다음 사람이 재발견.

구조 없는 문서를 AI가 읽으면

AI가 읽은 표: 항목 FEM IGA 형상함수 Lagrange NURBS 연속성 C0 Cp-1 메시세분화 h-refinement k-refinement
→ 열 수도, 값의 소속 열도 알 수 없음.

해결책은 다음 슬라이드의 Cline SR + API 서버 전략. 추가 GPU·신규 챗봇 불필요.

SLIDE 03

실행 방법: 4단계 파이프라인

작성자는 Word만 다룬다. JSON·PostgreSQL은 자동화가 담당.
STEP 1
소스 변환
PPT·Excel·PDF·MD를 직접 변환 (또는 Word로 옮긴 뒤 변환). PDF는 outline·표·메타를 추출하지만 정보 손실이 가장 크므로 표준 양식 권장.
›
STEP 2
표준 Word 작성
3원칙: Heading 스타일·그림 캡션·표 캡션. 전체 품질을 좌우하는 단계.
›
STEP 3
JSON 변환
python-docx 스크립트가 Word → 범용 JSON 자동 변환. 작성자는 스크립트 실행만.
›
STEP 4
PostgreSQL 적재
JSON을 DB에 저장. 태그·전문·Cline SR 연동 검색이 가능해진다.

Word를 중간 포맷으로 선택한 이유

학습 비용 0

PPT·Excel 작성자도 Word는 다룰 수 있음. 신규 도구 도입 불필요.

계층 구조 표현

Heading 1/2/3 스타일이 JSON 섹션 트리로 1:1 매핑.

변환 도구 성숙

python-docx·Pandoc 등 Word → 구조화 데이터 변환 도구 다수.

Excel 대용량 데이터 시트는 Word를 거치지 않고 JSON으로 직변환
SLIDE 04

Word 작성 3원칙

원칙 미준수 시 JSON 변환 품질 저하. 전체에서 가장 중요한 단계.

원칙 1 — Heading 스타일 적용

굵게 + 큰 글씨로 제목 흉내 금지.
Word의 Heading 1 / 2 / 3 스타일 필수.

이유:
변환 도구는 XML의 w:pStyle로 계층 판단. 굵은 텍스트(w:b)는 제목으로 인식 불가.

원칙 2 — 그림 캡션 (그림 아래)

캡션 위치는 그림 아래. "그림 1" 같은 번호만으로는 부족.

좋은 예:
Figure 1: 브라켓 von Mises 응력 분포 (하중 1,000N, 최대 250MPa, 빨간색 = 응력 집중)

이유: JSON 변환 후 이미지는 사라지고 캡션 텍스트만 남음. AI는 캡션으로 그림 내용 추정.

원칙 3 — 표 캡션 (표 위)

캡션 위치는 표 위. 본문에 "아래 표는 ~"처럼 녹이지 않는다.

좋은 예:
Table 1: IGA YAML 파라미터 목록 (KooRemapper v1.3.0)

이유: 표는 JSON의 headers + rows 배열로 저장. 캡션 없으면 표 주제를 AI가 모름.

나쁜 예 vs 좋은 예

✗ 나쁜 Word 문서

  • 제목: 굵게 + 폰트 크기 키움
  • 그림 캡션: "그림 1" (번호만)
  • 표 캡션: 없음, 본문에 녹임
  • 약어 풀어쓰기 누락 (IGA, NURBS 설명 없음)

✓ 좋은 Word 문서

  • 제목: Heading 1/2/3 스타일 적용
  • 그림 캡션: 조건·수치·색상 의미 포함
  • 표 캡션: 표 위에 "Table X: [설명]"
  • 약어 첫 등장 시 풀어쓰기: IGA(Isogeometric Analysis)
회사 표준 Word 템플릿(.dotx) 배포 시 Heading 스타일 적용 부담 감소
SLIDE 05

PPT 작성 표준 — Word와 동등한 구조화

PPT는 슬라이드 단위로 흩어져 있어 구조 추출이 어렵다. 슬라이드 제목·텍스트박스 번호·그림 캡션을 표준화하면 Word와 동일한 변환 품질을 얻는다.

PPT의 구조적 한계

슬라이드 단위 분리

본문이 슬라이드 경계로 끊긴다. 자연스러운 흐름이 형성되지 않는다.

텍스트박스 산개

좌우상하로 흩어진 텍스트박스. 등장 순서가 모호하다.

본문/보조 구별 불가

같은 슬라이드 안에서도 어느 박스가 본문이고 어느 박스가 보조인지 구분되지 않는다.

그림·표 식별 불가

그래픽 객체로 박힌 그림·표는 캡션 부재 시 내용을 식별할 수 없다.

표준 작성 원칙

원칙 1 — 슬라이드 번호 + 제목

모든 슬라이드는 Title placeholder를 사용한다. 제목은 1. 개요, 1.1 IGA란 무엇인가, 2. 작동 원리 형식으로 번호 + 텍스트로 작성한다.

이유:
Title placeholder는 Word의 Heading 1과 동일한 역할을 한다. 슬라이드 마스터(.potx)에서 Title을 강제하면 작성자 부담이 사라진다.

원칙 2 — 텍스트박스 라벨링

각 텍스트박스에 의미적 라벨을 명시한다: [목적], [방법], [결과], [결론]. 또는 박스 첫 줄에 목적:, 방법:처럼 소제목을 둔다.

이유:
좌→우, 상→하 배치 순서가 본문 흐름과 일치하도록 작성한다. 변환기는 라벨로 박스의 역할을 판단한다.

원칙 3 — 그림·표 캡션

그림 아래에 Figure N-M: 설명 (N=슬라이드 번호, M=슬라이드 내 순번). 표 위에 Table N-M: 설명.

이유:
캡션은 PowerPoint 텍스트박스로 작성하고 그림과 그룹화(Group)한다. 함께 이동·복사되어 위치 어긋남이 발생하지 않는다.

추가 권장 사항

슬라이드 노트 활용

발표 시 말로 보충하는 맥락은 슬라이드 노트(Speaker Notes)에 작성한다. 변환기가 노트를 본문 보조 단락으로 추출한다.

빌드/애니메이션 의존 금지

점진적 출현 같은 시각 효과로만 정보를 전달하지 않는다. 정적 상태에서 모든 정보가 읽혀야 한다.

표는 PowerPoint 표 객체

이미지로 붙여넣지 않는다. 셀 단위 추출이 가능해야 한다.

표준 .potx 템플릿 배포

슬라이드 마스터에 Title placeholder, 라벨드 텍스트박스, 캡션 스타일이 사전 정의된 템플릿을 사내 배포한다.

나쁜 예 vs 좋은 예

✗ 나쁜 PPT

  • 제목 없는 슬라이드 (메인 텍스트만 큼지막하게)
  • 좌우 텍스트박스 두 개 — 어느 게 먼저인지 모호
  • 그래프 이미지에 캡션 없음
  • 핵심 정보가 슬라이드 노트나 발표자 머릿속에만 있음

✓ 좋은 PPT

  • 2.1 IGA Trimmed NURBS Volume처럼 번호+제목 슬라이드 제목
  • 텍스트박스 첫줄에 목적:, 방법:, 결과: 명시
  • Figure 3-1: NURBS 박스 구조 (offset=4mm) 캡션
  • 핵심 내용이 슬라이드에 모두 적혀 있음
변환 흐름 일관성 — 같은 변환기·같은 JSON 스키마로 Word와 PPT를 동일하게 처리한다. PPT의 슬라이드는 Word의 Heading 1, 텍스트박스 라벨은 Heading 2, 캡션은 동일하게 해석된다.
표준 .potx 템플릿 + 작성 가이드를 배포하면 PPT도 Word와 동일한 변환 품질을 얻는다
SLIDE 06

Excel 작성 표준 — 표 데이터 + 데이터의 의미

Excel 시트 → DATA type JSON으로 직변환된다. 표 자체는 데이터, _META·_GLOSSARY 시트는 의미. AI가 데이터를 안다는 건 두 시트를 같이 읽는다는 것이다.

Excel의 구조적 한계

본문 흐름 부재

시트 단위로 분리되어 본문 흐름이 형성되지 않는다. Word·PPT 같은 서술 구조가 없다.

표 위치 임의성

표가 시트 임의 위치에 배치되면 헤더 자동 인식이 어려워진다.

서식 의미 소실

셀 병합·서식·색상으로 정보를 표현하면 변환 시 의미가 소실된다.

수식·차트 미캡처

수식은 계산 결과만 추출되고 차트·이미지는 변환되지 않는다.

6가지 작성 원칙

원칙 1 — 시트 상단 고정 (가장 중요)

표는 항상 A1부터 시작한다. 시트 중간에 애매하게 표를 띄우지 않는다.

이유:
1행 = 헤더, 2행부터 데이터. 표 위·옆에 빈 셀이나 메타 정보를 두면 헤더 자동 인식이 어려워진다.

원칙 2 — 헤더에 단위 명시

무게(kg), 응력(MPa), 시간(s)처럼 단위를 헤더에 포함한다.

이유:
변환기의 --infer-units 가 자동 분리한다: ("무게", "kg"). 같은 단위라도 명시하지 않으면 분석에서 단위 혼선이 발생한다.

원칙 3 — 셀 병합 금지

헤더 행은 절대 병합하지 않는다. 데이터 행도 같은 값을 반복 입력한다.

이유:
변환기는 병합된 셀 값을 모든 셀에 복제하지만 의도가 손실될 수 있다. 같은 값을 표현하려면 셀을 합치지 말고 반복 입력한다.

원칙 4 — 색상 의미 별도 컬럼화

"빨간 행 = 불량" 같은 색상 의미는 별도 컬럼(불량여부)으로 명시한다. 색상은 변환기에 전달되지 않는다. 조건부 서식 대신 데이터 값 자체로 표현한다.

원칙 5 — 1시트 1주제

하나의 시트에 한 가지 주제만 (한 종류의 측정 데이터). 같은 시트에 여러 표를 두지 않는다. 다른 주제는 다른 시트로 분리한다.

원칙 6 — 데이터 의미 명시 (★ 핵심)

표 자체는 숫자 행렬일 뿐. _META + _GLOSSARY 시트로 데이터의 의미·시험 방법·컬럼 정의를 Excel 자체에 기술한다.

이유:
"100 / 250 / 0.85" 만 보면 사람도 AI도 의미를 모른다. 컨텍스트가 없으면 데이터는 검색 불가능한 행렬로 남는다.

데이터 의미를 기술하는 3가지 방법

_META 시트 — 워크북·시트 컨텍스트

2열(key | value) 구조. 워크북 레벨 키(title/summary/tags/agents/domain)와 시트 레벨 키(sheet:Sheet1.method 등)를 모두 담는다.

title | 브라켓 하중 시험 (2026-04)
tags | 시험,브라켓,하중
sheet:Sheet1.method | KS B 0814
sheet:Sheet1.condition | 23±2℃, 50%RH
sheet:Sheet1.equipment | UTM-500

_GLOSSARY 시트 — 컬럼 의미 정의

4열(column | description | unit | dtype) 구조. 헤더와 매칭되는 컬럼별 의미·단위·자료형을 정의한다.

무게 | 시험 직전 측정 | g | float
단면적 | 시험부 단면적 | mm² | float
최대하중 | 시험 중 최대 하중 | N | float
파괴여부 | 파괴 발생 여부 | - | enum:Y/N

워크북 빌트인 속성 — 폴백

File → Info → Properties 의 Title, Subject, Comments, Keywords. _META가 없을 때만 사용되는 폴백.

우선순위 (낮음→높음):
빌트인 → 헤더 인라인 단위 →
_GLOSSARY → _META → CLI 인자

의미 없는 Excel vs 의미가 명시된 Excel

✗ 데이터만 있는 Excel

  • "100 / 250 / 0.85" — 단위와 의미를 알 수 없음
  • 이 시트가 무엇을 측정한 것인가? 모름
  • 측정 방법·시험 표준은? 모름
  • 측정 조건(온도·습도)은? 모름
  • "파괴여부" Y/N — Y 가 양품인가 불량인가? 모름
  • RAG 검색 시 데이터 매칭 불가능

✓ _META + _GLOSSARY 포함 Excel

  • _META.title — "브라켓 하중 시험 결과 (2026-04)"
  • sheet:Sheet1.method — "KS B 0814 표준 인장시험"
  • sheet:Sheet1.condition — "상온 23±2℃, 습도 50%RH"
  • _GLOSSARY.최대하중 — "시험 중 기록된 최대 하중 (N, float)"
  • tags=[시험,브라켓,하중] — RAG 검색 가능
  • AI가 "이 데이터는 무엇인가" 에 답할 수 있음

불규칙한 Excel 처리 방법

✗ 불규칙한 형태

  • 표가 시트 중간(예: B5)에서 시작
  • 시트 상단에 제목 셀과 메타 정보(작성자·날짜)가 박혀 있음
  • 데이터 사이에 빈 행이 들어가 있음
  • 여러 표가 한 시트에 좌우로 나란히 배치
  • 셀 병합으로 그룹 헤더 표현 (2행 헤더)
  • 차트·이미지가 데이터와 겹쳐 있음

✓ 변환기에 친화적인 형태

  • A1부터 헤더 시작
  • 빈 행 없이 연속된 데이터
  • 1시트 1주제, 시트 이름 = 데이터 주제
  • 단위는 헤더에 괄호로 명시
  • 색상·서식 의미는 별도 컬럼으로

불규칙 Excel 변환 옵션

--header-row N   # 헤더 행 위치 명시 (기본 1)
--start-cell A5   # 표 시작 셀 명시 (불규칙 시트용)
--mode combined    # 모든 시트를 하나의 record로 묶음
--skip-empty     # 빈 시트 자동 무시
--infer-units    # 헤더의 (단위) 자동 분리
--meta-sheet _META    # 워크북·시트 컨텍스트 시트 이름
--glossary-sheet _GLOSSARY# 컬럼 의미 정의 시트 이름
# 변환 후 검수 필수 — 변환기 경고 로그 확인
표준 배포 전략 — 일반 사용자가 Excel을 자유롭게 만들면 변환 손실이 크다. 표 작성 표준을 사전 배포하고, 표준을 따르지 않는 기존 자료는 --start-cell·--header-row 옵션으로 보정한다.
표 형식 5원칙 + 의미 명시 1원칙 = 6원칙. _META·_GLOSSARY 가 있어야 데이터가 검색 가능한 자산이 된다.
SLIDE 07

범용 JSON 스키마

AI가 문서를 이해하려면 텍스트 덩어리가 아닌 구조화된 형태가 필요.

최상위 6개 키 구조

"schema_version": "1.0", ← 호환성 관리
"meta": { ... }, ← 문서 신원증명서
"toc": [ ... ], ← AI용 목차 (가벼운 구조 파악)
"sections": [ ... ], ← 계층 트리 (본문)
"figures": [ ... ], ← 그림 캡션 (이미지 대역)
"tables": [ ... ] ← headers + rows 배열

meta.summary — 가장 중요한 필드

RAG 시스템은 수백 개 문서 중 관련 문서를 추릴 때 전체를 읽지 않는다. summary만 읽어 질문 관련도를 판단. summary 누락 시 RAG 1차 필터 작동 불가.

각 키가 존재하는 이유

키없으면 일어나는 일
meta.doc_id버전이 다른 같은 이름 파일을 AI가 구분 못함
meta.tags"IGA 관련 문서 찾아줘" 쿼리가 느리고 부정확
toc"이 문서 구조가 어떻게 되나" 질문에 sections 전체를 읽어야 함
sections 트리1장의 하위 항목이 어디까지인지 AI가 알 수 없음
figures.captionAI는 이미지를 볼 수 없으므로 그림 내용을 전혀 모름
tables 배열"FEM의 연속성은?" 질문에 답할 수 없음

표가 평문 vs 구조화 배열

평문: "항목 FEM IGA 연속성 C0 Cp-1" → 열 구분 불가

"headers":["항목","FEM","IGA"], "rows":[["연속성","C0","Cp-1"]] → 정확한 비교 가능

직원이 JSON을 직접 다룰 일 없음. 변환 스크립트가 Word에서 자동 생성.
SLIDE 08

PostgreSQL로 데이터 축적

JSON 파일을 파일 시스템에만 두면 단순 보관. PostgreSQL에 적재하면 검색·AI 연동 가능.

왜 PostgreSQL인가

JSONB

JSON 이진 저장·인덱싱. 내부 필드를 SQL로 직접 쿼리.

전문 검색

한국어 포함 다국어 Full-Text Search 내장.

pgvector

AI 벡터 임베딩 저장·유사도 검색. RAG 시스템 핵심.

핵심 테이블 구조

테이블역할주요 컬럼
records 레코드 전체 저장 record_id, tags[], classification, status, capabilities[], content JSONB
record_sections 섹션 단위 검색용 record_id, section_id, content, fts_vector, embedding VECTOR(384)
record_attachments 9종 첨부 일반화 record_id, kind(figure/document/spreadsheet/...), caption, file_path
api_keys X-API-Key 인증 key_hash(sha256), label, scopes[], created_at
agent_scope 에이전트-레코드 매핑 agent_type, record_id, priority

capabilities는 본문 모양에서 자동 산출(sections/blocks/tables/figures/attachments/embeddings). classification·status·capabilities에 GIN 인덱스.

대표 쿼리 3종

태그 검색

IGA·LS-DYNA 태그가 붙은 문서만 필터링.

WHERE tags @> ARRAY['IGA','LS-DYNA']

전문 검색

섹션 내용에서 "NURBS 제어점" 키워드 포함 섹션 검색.

WHERE fts_vector @@ to_tsquery('NURBS & 제어점')

시맨틱 검색 (pgvector)

질문 의미와 가장 유사한 섹션을 찾아 AI 프롬프트에 전달.

ORDER BY embedding <=> $1 LIMIT 5
PostgreSQL: 오픈소스, 라이선스 비용 없음, Docker 한 줄로 설치

JSON 스키마 상세 규칙

[json_schema_rules.md] v1.0 — 7개 최상위 키 + 전역 고유 ID 체계로 모든 레코드를 일관되게 표현. attachments는 9종을 포괄.

7개 최상위 키 (모두 필수)

{
  "schema_version": "1.0",
  "meta": {...}, // 신원증명서 + 분류·상태·관계
  "toc": [...], // 가벼운 목차
  "sections": [...], // 본문 (blocks 트리)
  "attachments": [...], // 9종 첨부 (figure/document/...)
  "tables": [...], // 표 데이터 (headers+rows)
  "sources": [...] // CAD 등 외부 파일 참조
}

attachments — figure 일반화

9종 kind: figure / document / spreadsheet / media / archive / cad / drawing / data / other. 캡션은 모든 kind에 필수. 파일은 /attachments/{record_id}/{filename}으로 정적 서빙.

meta 12개 신규 필드

classification·status·domain·subject_keywords·source_system·language·parent_record_id·derivation·capabilities·quality_score·valid_from·valid_until. 다음 슬라이드에서 상세.

전역 고유 ID 체계

ID 종류형식예시
문서{사업부}-{팀}-{연도}-{6자리}HE-CAE-2026-000001
섹션{숫자}.{숫자}1.2, 2.3.1
그림{doc_id}-F{3자리}...-F001
표{doc_id}-T{3자리}...-T001
외부 소스{doc_id}-S{3자리}...-S001

sources — CAD 파일 동일성 판정

같은 CAD 파일이 여러 문서에서 참조될 때 동일성 판정용 메타데이터를 함께 저장.

  • file_path, file_name, modified, size_bytes
  • hash_sha256 — 정밀 판정 (권장)
  • type: MCAD / ECAD / DRAWING / SIM / DOC / OTHER

모든 ID가 전역 고유하므로, PostgreSQL에서 그림·표 참조 시 출처 문서·부서를 즉시 추적 가능.

메타데이터 보강 + 분류 API

일반화된 데이터에 12개 메타 필드 추가. 같은 데이터를 모양별·분류별로 슬라이스해서 가져온다.

메타데이터 12 신규 필드

분류 · 상태

  • classification — public / internal / confidential / restricted
  • status — draft / review / approved / deprecated

도메인 · 언어

  • domain, subject_keywords[]
  • source_system, language

관계 (parent/child)

  • parent_record_id
  • derivation — original / extracted / aggregated / translated

품질 · 유효기간

  • quality_score
  • valid_from, valid_until

capabilities[] — 가장 중요

본문 모양에서 자동 산출. 가능한 값: sections, blocks, tables, figures, attachments, embeddings. 분류 API의 핵심 인덱스.

분류 API — 같은 풀, 다른 모양

# 모양별 view (capabilities 기반)
GET /api/views/hierarchical # sections+blocks 보유만
GET /api/views/tabular # tables 보유만
GET /api/views/generalized # 전체, content 제외 slim

# 필터 + 분류 조합
GET /api/records?capabilities=sections
  &classification=approved

# 서브리소스 (record 단위 모양 추출)
GET /api/records/{id}/sections
GET /api/records/{id}/tables
GET /api/records/{id}/figures
GET /api/records/{id}/blocks

설계 의도

모든 레코드는 같은 일반화 스키마로 적재. 소비자(에이전트)는 필요한 모양만 골라 읽는다.

  • 표 분석 에이전트 → /views/tabular만
  • 위키 렌더러 → /views/hierarchical
  • 대시보드 카탈로그 → /views/generalized

capabilities 기반 자동 라우팅

레코드를 적재하면 본문 모양을 보고 capabilities가 자동 채워짐. 작성자는 메타를 손대지 않아도 분류 API에 자동 노출.

같은 일반화 데이터 풀에서 에이전트별로 필요한 모양만 효율적으로 가져온다. 새 모양이 필요하면 view 1개를 추가하면 된다.

capabilities는 ingest 시 자동 산출. classification·status·valid_until은 거버넌스용 필수 메타.
SLIDE 11

sections.blocks — 순서 보존

본문을 평탄한 텍스트가 아닌 블록 배열로 구성. 표·그림이 원본 위치에 박혀 있어 AI가 맥락 그대로 읽음.

blocks 배열이 필요한 이유

✗ 평탄 텍스트

"content": "FEM과 IGA의 차이점:
IGA는 특히 얇은 쉘에서..."

표 위치 소실. "이 도입 문장 뒤에 비교표가 있었다"는 맥락 누락.

✓ blocks 배열

"blocks": [
 {"type":"paragraph", "text":"FEM과 IGA의 차이점:"},
 {"type":"table", "ref":"...-T001"},
 {"type":"paragraph", "text":"IGA는 특히..."}
]

표 위치가 본문 흐름 안에 명시. AI가 도입 → 표 → 결론 그대로 읽음.

5가지 블록 타입

paragraph

일반 단락. 본문 텍스트의 기본 단위.

code

등폭 글꼴 블록. ASCII 다이어그램·LS-DYNA 카드·YAML 등.

table

ref로 표 데이터 지시. 데이터는 tables[]에 저장.

figure

ref로 그림 지시. 데이터는 figures[]에 저장.

list_item

목록 항목. marker(•/1.)와 함께 표기.

이 구조가 있어야 AI가 "이 표는 이 단락의 결론을 보여주는 표"라는 의미적 연결을 인식.

data ≠ context. 데이터 추출과 맥락 보존은 별개의 문제.

ALTER TABLE record_sections
  ADD COLUMN embedding VECTOR(384);
CREATE INDEX ON record_sections
  USING ivfflat (embedding vector_cosine_ops);

EmbeddingProvider ABC

SQLite 폴백

SLIDE 15

휴리스틱 헤딩 감지

Heading 스타일이 없는 레거시 문서용 3단계 자동 감지. 단순 번호 목록과 섹션 헤딩을 구별.

① 정규식 패턴

본문 단락 중 1., 1.1, 1.1.1 형태(끝점 유무 모두) 후보 수집.

^(\d+(?:\.\d+){0,2})\.?\s+(.+)$

② Sub-heading 기반 확정

level 2 후보(1.1)는 무조건 헤딩으로 확정.
level 1 후보(1.)는 해당 prefix의 level 2가 뒤에 등장하는 경우에만 헤딩 확정.

→ 단순 번호 목록과 헤딩 자동 구별

③ 시퀀스 규칙 (꼬리 보완)

마지막 확정 헤딩(10) 다음 11, 12는 sub-heading 없어도 자동 추가.

→ 부록 직전 마지막 절(보통 sub 없음) 처리

단순 번호 목록 vs 헤딩 — 자동 구별

✓ 헤딩으로 인식

2. 작동 원리 ← level 1
  2.1 Trimmed NURBS Volume 개념 ← level 2 등장
  2.2 생성 흐름
  2.3 ID 할당 규칙

"2."가 sub 2.1, 2.2, 2.3을 가지므로 헤딩 확정

✗ 단순 번호 목록 (헤딩 아님)

KooRemapper의 iga 명령은 다음 과정을 자동화한다:
1. Bounding Box 계산
2. Offset 확장
3. NURBS 제어점 생성
... (sub 없음)

sub-heading 없음 → 헤딩 확정 안 됨, paragraph 유지

이 휴리스틱으로 기존 문서를 그대로 변환 가능. 표준 양식 강제 이전에 기존 자료부터 활용 가능한 상태로 전환하는 것이 도입 전략의 출발점.

작성 시점부터 [제목 1/2/3] 스타일 적용 시 휴리스틱 없이 더 정확한 결과

AI가 JSON만 보고 답한다

변환된 JSON 하나로 원본 Word 없이 도메인 질문 응답 가능. RAG·Cline SR의 실제 동작 방식.

Q&A 시뮬레이션

Q1. "이 문서가 다루는 핵심 주제는?"

A: IGA(등기하해석)를 KooRemapper로 자동화하는 LS-DYNA R12+ 매뉴얼. NURBS 기반 Trimmed Volume 방식으로 FE solid mesh를 IGA로 변환하는 절차·YAML 문법·생성 카드 설명.

📍 출처: meta.title + meta.doc_type + sections[0~2]

Q2. "FEM과 IGA의 형상 함수 차이는?"

A: FEM(HEX8/TET4)은 Lagrange 다항식, IGA(NURBS)는 NURBS 기저함수. 형상 표현은 FEM은 근사(절점 보간), IGA는 정확(CAD와 동일).

📍 출처: sections[1.1].blocks → table ref T001 → tables[].rows[0]

Q3. "다항식 차수는 어떻게 선택하나?"

A: 평판/쉘은 pr=2, ps=2, pt=1 (두께 방향 선형). 3D 볼륨 고정밀은 전 방향 2차. 빠른 초기 계산은 모두 1차. 차수 1→C0, 2→C1, 3→C2 연속성.

📍 출처: sections[5.1] + tables T004 + sections[9.3] 종합

Q4. "어느 부서에서 작성했고 누가 봐야 하나?"

A: HE-CAE 부서 작성. 내용상 IGA 해석 담당자·KooRemapper 코드 작업자가 참조. (agent_scope 마커 누락 — 작성 보완 필요)

📍 출처: meta.department (agent_scope 비어 있음 → 본문 추론)

AI의 JSON 탐색 4단계

1
meta.summary 1차 필터
수백 개 문서 중 관련 후보 선별 (RAG)
›
2
toc로 구조 파악
52개 항목으로 위치 인지
›
3
관련 섹션 blocks 정독
원본 등장 순서대로 paragraph·code·table 재구성
›
4
table ref 조회
block의 ref id로 tables[]에서 headers·rows 직접 인용

이 4단계가 PostgreSQL + Cline SR 환경에서 그대로 동작. JSON 품질만 확보하면 추가 시스템 없이 사업부 도메인 지식이 AI에 전달.

잘 작성된 JSON 1개 = 학습된 AI 도메인 지식 1건
