SLIDE 02

데이터 구조화 = 미래의 GPU 비용 절감

큰 LLM = 비정형 데이터 추론 복원 가능, GPU·RAM·시간 비용 큼. 사전 구조화 → 작은 모델(3B~7B)도 검색·요약·QA·비교 가능. 모델 성능 향상 < 데이터 구조 정비.

두 접근의 비용 구조

큰 모델 중심

비정형 데이터 그대로 투입 → 추론으로 의미 복원.

  • VRAM 요구량 증가 → 대형 GPU 필수
  • RAM·연산·추론 시간 ↑
  • 운영 비용 누적 — 사용량 비례
  • 폐쇄망·온프레미스 부담 큼
  • 모델 교체 시마다 재투자

구조화 데이터 중심

의미·계층·관계 사전 인코딩 → 데이터가 추론 부담 흡수.

  • 작은 모델(3B~7B) 안정 운영
  • GPU 의존도 ↓ — CPU 추론 가능
  • 온프레미스·폐쇄망 적합
  • 유지비 낮음, 수평 확장 용이
  • 모델 교체에도 데이터 자산 유지

SLIDE 05

Word 작성 3원칙

원칙 미준수 → JSON 변환 품질 저하. 전체 파이프라인 최중요 단계.

원칙 1 — Heading 스타일 적용

굵게 + 큰 글씨로 제목 흉내 금지.
Word Heading 1 / 2 / 3 스타일 필수.

이유:
변환 도구는 XML의 w:pStyle로 계층 판단. 굵은 텍스트(w:b) → 제목 인식 불가.

원칙 2 — 그림 캡션 (그림 아래)

위치 = 그림 아래. "그림 1" 번호만으로 부족.

좋은 예:
Figure 1: 브라켓 von Mises 응력 분포 (하중 1,000N, 최대 250MPa, 빨간색 = 응력 집중)

이유: JSON 변환 후 이미지 소실, 캡션 텍스트만 잔존. AI는 캡션으로 그림 내용 추정.

원칙 3 — 표 캡션 (표 위)

위치 = 표 위. 본문에 "아래 표는 ~" 식 융합 X.

좋은 예:
Table 1: IGA YAML 파라미터 목록 (KooRemapper v1.3.0)

이유: 표 = JSON headers + rows 배열. 캡션 부재 시 표 주제 식별 불가.

나쁜 예 vs 좋은 예

✗ 나쁜 Word 문서

  • 제목: 굵게 + 폰트 크기 키움
  • 그림 캡션: "그림 1" (번호만)
  • 표 캡션: 없음, 본문에 녹임
  • 약어 풀어쓰기 누락 (IGA, NURBS 설명 없음)

✓ 좋은 Word 문서

  • 제목: Heading 1/2/3 스타일 적용
  • 그림 캡션: 조건·수치·색상 의미 포함
  • 표 캡션: 표 위에 "Table X: [설명]"
  • 약어 첫 등장 시 풀어쓰기: IGA(Isogeometric Analysis)

PPT 작성 표준 — Word와 동등한 구조화

PPT = 슬라이드 단위 분산 → 구조 추출 곤란. 슬라이드 제목·텍스트박스 번호·그림 캡션 표준화 → Word 동등 변환 품질.

PPT의 구조적 한계

슬라이드 단위 분리

본문 = 슬라이드 경계로 단절. 흐름 형성 불가.

텍스트박스 산개

좌우상하 분산 텍스트박스 → 등장 순서 모호.

본문/보조 구별 불가

동일 슬라이드 내 본문 박스 vs 보조 박스 식별 불가.

그림·표 식별 불가

그래픽 객체화 그림·표 — 캡션 부재 시 내용 식별 불가.

표준 작성 원칙

원칙 1 — 슬라이드 번호 + 제목

모든 슬라이드 Title placeholder 사용 필수. 제목 형식 = 1. 개요, 1.1 IGA란 무엇인가, 2. 작동 원리 (번호 + 텍스트).

이유:
Title placeholder ≡ Word Heading 1. .potx 마스터에서 Title 강제 → 작성자 부담 0.

원칙 2 — 텍스트박스 라벨링

각 텍스트박스에 의미 라벨 명시 — [목적], [방법], [결과], [결론]. 또는 박스 첫 줄에 목적:, 방법: 형식의 소제목.

이유:
좌→우, 상→하 배치 = 본문 흐름 일치. 변환기는 라벨로 박스 역할 판단.

원칙 3 — 그림·표 캡션

그림 아래 Figure N-M: 설명 (N=슬라이드 번호, M=슬라이드 내 순번). 표 위 Table N-M: 설명.

이유:
캡션 = PowerPoint 텍스트박스 + 그림과 그룹화(Group). 동시 이동·복사 → 위치 어긋남 방지.

추가 권장 사항

슬라이드 노트 활용

구두 보충 맥락 → 슬라이드 노트(Speaker Notes). 변환기가 노트를 본문 보조 단락으로 추출.

빌드/애니메이션 의존 금지

점진적 출현 등 시각 효과 단독 정보 전달 X. 정적 상태에서 모든 정보 가독.

표 = PowerPoint 표 객체

이미지 붙여넣기 X → 셀 단위 추출 가능 형태.

표준 .potx 배포

슬라이드 마스터 = Title placeholder + 라벨드 텍스트박스 + 캡션 스타일 사전 정의. 사내 배포.

6가지 작성 원칙

원칙 1 — 시트 상단 고정 (가장 중요)

표 시작 = A1 고정. 시트 중간 애매 배치 X.

이유:
1행 = 헤더, 2행~ = 데이터. 표 위·옆 빈 셀·메타 → 헤더 자동 인식 곤란.

원칙 2 — 헤더에 단위 명시

단위 헤더 포함 — 무게(kg), 응력(MPa), 시간(s).

이유:
--infer-units 자동 분리 → ("무게", "kg"). 단위 미명시 → 분석 단위 혼선.

원칙 3 — 셀 병합 금지

헤더 행 병합 절대 금지. 데이터 행 동일 값 반복 입력.

이유:
변환기는 병합 셀 값을 각 셀에 복제 — 그룹 의도 소실. 동일 값 표현 = 셀 병합 X, 반복 입력 O.

원칙 4 — 색상 의미 별도 컬럼화

"빨간 행 = 불량" 등 색상 의미 → 별도 컬럼(불량여부) 명시. 색상 = 변환기 미전달. 조건부 서식 대신 데이터 값 자체로 표현.

원칙 5 — 1시트 1주제

1시트 = 1주제 (단일 측정 데이터 종류). 동일 시트 다중 표 X. 다른 주제 → 별도 시트 분리.

원칙 6 — 데이터 의미 명시 (★ 핵심)

표 = 숫자 행렬일 뿐. _META + _GLOSSARY → 데이터 의미·시험 방법·컬럼 정의를 Excel 자체에 기술.

이유:
"100 / 250 / 0.85" 단독 → 사람·AI 의미 식별 불가. 컨텍스트 부재 → 검색 불가 행렬.

데이터 의미를 기술하는 3가지 방법

_META 시트 — 워크북·시트 컨텍스트

2열(key | value) 구조. 워크북 레벨 키(title/summary/tags/agents/domain) + 시트 레벨 키(sheet:Sheet1.method 등) 동시 수용.

title | 브라켓 하중 시험 (2026-04)
tags | 시험,브라켓,하중
sheet:Sheet1.method | KS B 0814
sheet:Sheet1.condition | 23±2℃, 50%RH
sheet:Sheet1.equipment | UTM-500

_GLOSSARY 시트 — 컬럼 의미 정의

4열(column | description | unit | dtype) 구조. 헤더 매칭 컬럼별 의미·단위·자료형 정의.

무게 | 시험 직전 측정 | g | float
단면적 | 시험부 단면적 | mm² | float
최대하중 | 시험 중 최대 하중 | N | float
파괴여부 | 파괴 발생 여부 | - | enum:Y/N

워크북 빌트인 속성 — 폴백

File → Info → Properties의 Title, Subject, Comments, Keywords. _META 부재 시 폴백.

우선순위 (낮음→높음):
빌트인 → 헤더 인라인 단위 →
_GLOSSARY → _META → CLI 인자

의미 없는 Excel vs 의미가 명시된 Excel

✗ 데이터만 있는 Excel

  • "100 / 250 / 0.85" — 단위와 의미를 알 수 없음
  • 이 시트가 무엇을 측정한 것인가? 모름
  • 측정 방법·시험 표준은? 모름
  • 측정 조건(온도·습도)은? 모름
  • "파괴여부" Y/N — Y 가 양품인가 불량인가? 모름
  • RAG 검색 시 데이터 매칭 불가능

✓ _META + _GLOSSARY 포함 Excel

  • _META.title — "브라켓 하중 시험 결과 (2026-04)"
  • sheet:Sheet1.method — "KS B 0814 표준 인장시험"
  • sheet:Sheet1.condition — "상온 23±2℃, 습도 50%RH"
  • _GLOSSARY.최대하중 — "시험 중 기록된 최대 하중 (N, float)"

"toc": [ ... ], ← AI용 목차 (가벼운 구조 파악)

summary = RAG 1차 필터, agent_hints = 사용 가이드.

SLIDE 16

AI는 데이터를 어떻게 보는가 — 7대 구조화 원칙

RAG 품질 = 구조의 질 (텍스트 양 X). 작은 모델일수록 헤딩 계층·메타·관계 신호 의존도 ↑. 7원칙 + 4핵심 신호.

4가지 핵심 신호 — AI 가 데이터를 강하게 다루는 조건

계층 (Hierarchy)

트리 구조 → chunk 경계 보존.

관계 (Relation)

주장-근거, 그래프 엣지, 버전 체인.

의미 (Semantics)

메타·태그·name → 텍스트 객체화.

추적성 (Traceability)

버전 부모, decision 체인, evidence 근거.

7대 구조화 원칙 — 사업부 문서가 AI 자산이 되기 위한 조건

1. Heading Tree (최우선)

프로젝트→목적→입력→결과→의사결정. 계층 부재 → chunk 경계 붕괴, retrieval 품질 ↓, hallucination ↑. 작은 모델일수록 의존도 절대적.

2. Semantic Chunk

한 chunk = 의미적으로 완결된 정보. Claim/Evidence/Cause/Action 같은 sub-구조가 양질 chunk 형성. 표 반만 자른 chunk·다른 주제 섞임 = 0점.

3. Metadata

project·domain·solver·tags·date·author. retrieval 핵심 신호. 메타 부재 → 작은 모델이 문맥 추론 전담.

4. Claim → Evidence

주장 → 근거 → 결론 관계. 보고서 자동화·QA·요약·의사결정 추적성 모두 강함. ## Claim / ## Evidence / ## Conclusion 패턴.

5. Entity

Part ID·Material·Case Name·DOE ID·Node Set·Failure Mode 등 명명된 객체. graph화·cross-reference 가능 조건.

6. Graph 관계

프로젝트 ↔ DOE ↔ Simulation Result ↔ Failure ↔ Decision ↔ Field Claim. "이 failure mode와 가장 관련 높은 DOE는?" 질의 가능.

7. Time-series / Version

설계안 v1 → DOE 결과 → 문제 발생 → v2 → 재해석. parent·changed_part·reason → 설계 변경 이력 추적.

이전 슬라이드(JSON 스키마·sections.blocks·메타데이터·변환기) = 7원칙 자동화 구조. 작성자 = Heading 구조 + 메타만 담당, 나머지는 변환기 채움.

텍스트 양 ≠ 데이터 질 — 4신호(계층·관계·의미·추적성) 보존 시 RAG 작동
